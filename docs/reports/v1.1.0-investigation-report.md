## 一、OpenCode 官方文档核心要点

### 1. **工具系统 (Tools)**

**官方机制**：
- 工具通过 `permission` 配置控制（`allow` / `ask` / `deny`）
- 支持通配符模式匹配（如 `mymcp_*`）
- 自定义工具必须放在 `.opencode/tools/` 或 `~/.config/opencode/tools/`
- 工具定义使用 `tool()` 辅助函数，提供类型安全
- **文件名即工具名**，单文件多工具导出格式为 `<filename>_<exportname>`

**关键发现**：
- 自定义工具可以**覆盖内置工具**（如果同名）
- 工具接收 `context` 参数：`{ agent, sessionID, messageID, directory, worktree }`
- 工具可以用任何语言编写（TypeScript/JavaScript 只是定义层）

### 2. **代理系统 (Agents)**

**官方机制**：
- **主代理 (Primary)**：用户直接交互，Tab 键切换
- **子代理 (Subagent)**：主代理调用或 `@` 提及
- 代理配置支持 JSON 和 Markdown 两种方式
- 代理可以覆盖全局权限、工具、模型

**关键发现**：
- `mode` 选项：`primary` / `subagent` / `all`
- `hidden: true` 可以隐藏子代理（不在 `@` 菜单显示）
- `permission.task` 控制代理可以调用哪些子代理（支持 glob 模式）
- **用户始终可以通过 `@` 直接调用任何子代理**，即使权限拒绝

### 3. **技能系统 (Skills)**

**官方机制**：
- 技能通过 `SKILL.md` 定义，必须包含 YAML frontmatter
- 必填字段：`name`、`description`
- 技能名称规则：1-64 字符，小写字母数字，单连字符分隔
- 技能通过 `skill` 工具按需加载

**关键发现**：
- 技能搜索路径：`.opencode/skills/`、`~/.config/opencode/skills/`、`.claude/skills/`、`.agents/skills/`
- 技能权限通过 `permission.skill` 控制（支持模式匹配）
- `tools.skill: false` 可以完全禁用技能功能

### 4. **命令系统 (Commands)**

**官方机制**：
- 命令通过 `.opencode/commands/` 中的 Markdown 文件定义
- 支持参数：`$ARGUMENTS`、`$1`、`$2` 等
- 支持 Shell 输出注入：`!`command``
- 支持文件引用：`@filename`

**关键发现**：
- 命令可以指定 `agent`、`model`、`subtask` 选项
- `subtask: true` 强制命令作为子代理运行
- 自定义命令可以**覆盖内置命令**

### 5. **权限系统 (Permissions)**

**官方机制**：
- 权限解析为 `allow` / `ask` / `deny`
- 支持对象语法进行细粒度控制（如 `bash: { "*": "ask", "git *": "allow" }`）
- **最后匹配的规则优先**
- 支持主目录展开（`~` 或 `$HOME`）

**关键发现**：
- `external_directory` 权限控制工作目录外的路径访问
- `doom_loop` 权限：同一工具调用重复 3 次时触发
- 代理权限会与全局配置合并，代理规则优先

### 6. **规则系统 (Rules)**

**官方机制**：
- 规则通过 `AGENTS.md` 文件定义
- 支持项目级（`.opencode/AGENTS.md`）和全局级（`~/.config/opencode/AGENTS.md`）
- 支持 Claude Code 兼容（`CLAUDE.md`、`~/.claude/CLAUDE.md`）

**关键发现**：
- 优先级：本地 `AGENTS.md` > 全局 `AGENTS.md` > `~/.claude/CLAUDE.md`
- 可以通过 `instructions` 字段引用外部文件（支持 glob 模式）
- 远程 URL 获取超时 5 秒

## 二、现有实现问题分析（基于真实代码检查）

### 问题 1：Auto-Review 机制冲突 ⚠️ **高优先级**

**代码位置**：
- `src/hooks/todo-continuation/index.ts` 第 32-66 行的 `REVIEW_PROMPT`

**现状**：
- 要求主 agent 调用 `task(subagent_type="oracle")` 启动子 agent 会话
- 第 36 行：`1. **Spawn @oracle** — Use the task tool: task(subagent_type="oracle"...)`

**问题**：
- 违背"主 agent 优先"原则
- 启动子 agent 会话会创建新上下文，增加 token 成本
- 缓存失效，降低性能
- **当前上下文已达 140K tokens，接近 500K 危险阈值**

**根因**：
- 设计时混淆了"子 agent 作为工具"和"子 agent 作为会话"
- @oracle 应该作为**内部 checklist**，而不是独立会话

**官方文档对照**：
- 官方文档中，子 agent 通过 `task` 工具调用是合法的
- 但我们的架构目标是"主 agent 优先"，应该避免频繁启动子会话

### 问题 2：save_plan 工具 Description 不够明确 ⚠️ **高优先级**

**代码位置**：
- `src/tools/save-plan.ts` 第 8-10 行

**现状**：
```typescript
description: `Save a planner-created markdown plan to .opencode/extendai-lab/plans/ and return the real saved path.

Use this when the planner/prometheus agent has enough information to create an executable plan. Do not claim a plan was saved unless this tool returns success.`
```

**问题**：
- Description 只说了"保存到 .opencode/extendai-lab/plans/"
- **没有明确告诉 AI：必须调用这个工具，而不是输出到对话框**
- 缺少关键指令："CRITICAL: Use this tool to save the plan. Do NOT output the plan in the conversation."
- AI 有时会直接在对话框输出计划内容，而不是调用工具

**根因**：
- 工具 description 不够强制性
- 缺少"禁止行为"的明确说明
- 反复注入路径提示导致上下文膨胀（token 浪费）

**正确的设计理念**（用户指出）：
- 应该让 AI **决定调用工具**，通过清晰的 description 引导
- 而不是反复注入路径提示（浪费 token，即使有缓存命中）
- 工具 description 应该一次性说清楚所有规则

**官方文档对照**：
- 官方文档强调工具 description 应该清晰、具体
- 自定义工具的 description 是 LLM 决策的唯一依据

### 问题 3：Start Work 命令在新窗口的上下文隔离问题 ⚠️ **高优先级**

**代码位置**：
- `src/hooks/start-work/index.ts` 第 25-98 行
- `src/commands/start-work-template.ts` 第 1-63 行
- `src/index.ts` 第 1390-1393 行（hook 调用）

**现状**：
- `/ol-start-work` 命令使用**双重触发机制**：
  1. **Hook 触发**（第 1390 行）：`startWorkHook.handleCommandExecuteBefore` 读取计划文件，构建上下文，注入到 `output.parts`
  2. **Template 触发**（第 1193-1198 行）：`START_WORK_TEMPLATE` 提供执行指导
- Hook 检测逻辑正常（`start-work/index.ts` 第 34 行）

**问题**：
- 用户在**新窗口**执行 `/ol-start-work` 时，AI 无法正确执行
- 新窗口是**隔离上下文**（这是设计意图，避免上下文过重）
- 但新窗口可能缺少必要的 session 状态或 boulder.json 状态

**根因**：
- 新窗口的 session 上下文可能没有正确初始化
- boulder.json 状态可能没有跨窗口传递
- 或者 AI 在新窗口中没有正确理解 hook 注入的上下文

**设计意图**（用户说明）：
- Plan 做好后**应该新开会话**执行，因为：
  1. 隔离上下文（planning 和 execution 分离）
  2. 原会话上下文可能已经很重
- Checkpoint 机制也是类似设计

**官方文档对照**：
- 官方命令系统支持 `command.execute.before` hook
- Hook 可以修改 `output.parts` 注入上下文
- 但跨窗口的状态传递需要额外处理

### 问题 4：MCP 多窗口共享问题 🔧 **中优先级**

**现状**：
- stdio 传输 1:1 限制
- 第一个窗口正常，后续窗口失败

**根因**：
- MCP stdio 协议本身的限制
- 共享逻辑尝试复用进程但失败

**官方文档对照**：
- 官方文档没有提到多窗口共享
- 建议方案：移除共享逻辑，每个窗口独立启动（或使用随机端口）

### 问题 5：Skill 读取问题 🔧 **中优先级**

**现状**：
- 技能有时读取失败

**根因**：
- 可能是路径搜索逻辑有问题
- 或者 frontmatter 解析错误

**官方文档对照**：
- 技能搜索路径：`.opencode/skills/`、`~/.config/opencode/skills/`、`.claude/skills/`、`.agents/skills/`
- 必须包含 `name` 和 `description` 字段
- 技能名称必须符合正则：`^[a-z0-9]+(-[a-z0-9]+)*$`

### 问题 6：Delete Guard 未触发 🔧 **中优先级**

**现状**：
- 删除保护已注册但可能未触发

**根因**：
- 工具名称不匹配
- OpenCode 可能使用不同的工具名称

**官方文档对照**：
- 官方权限系统支持 bash 命令的细粒度控制
- 可以用 `bash: { "rm *": "deny" }` 来阻止删除命令

## 三、关键发现和建议（基于真实代码分析）

### 1. **命令系统的双重触发机制** ✅ **已正确实现**

**代码证据**：
- `src/index.ts` 第 1193-1229 行：命令注册（Template-based）
- `src/index.ts` 第 1342-1415 行：命令拦截（Hook-based）

**两种触发方式**：

**A. Prompt 触发（Template-based）**
- 每个命令有一个 `template` 字段
- 当用户输入命令时，OpenCode 将 `template` 作为 prompt 注入给 LLM
- 例如：`/ol-start-work` 的 template 是 `START_WORK_TEMPLATE`（第 1193-1198 行）

**B. 代码触发（Hook-based）**
- 命令通过 `command.execute.before` hook 拦截
- Hook 直接修改 `output.parts`，注入上下文信息
- 例如：`startWorkHook.handleCommandExecuteBefore`（第 1390-1393 行）

**关键点**：`/ol-start-work` 命令**同时使用两种机制**：
1. Hook 先执行，读取计划文件，构建上下文（`start-work/index.ts` 第 79-89 行）
2. Template 后注入，提供执行指导（`start-work-template.ts`）

**结论**：命令系统实现正确，问题不在检测机制本身。

### 2. **插件注册机制** ✅ **符合官方规范**

**官方要求**：
- 工具、代理、命令、技能都应该通过 SDK 注册
- **不能**依赖用户手动修改 `opencode.json`
- 插件卸载后不应留下残留配置

**我们的实现**（代码证据）：
- ✅ 工具通过 `tool()` 辅助函数注册（`src/index.ts` 第 760-773 行）
- ✅ 代理通过 `config` hook 注册（第 777-1230 行）
- ✅ 命令通过 `registerCommandIfMissing` 注册（第 232-240 行）
- ✅ 技能通过 `opencodeConfig.skills.paths` 注册（第 1116-1142 行）

**结论**：插件注册机制完全符合官方规范。

### 3. **上下文压力和 Checkpoint 机制** ⚠️ **需要增强**

**当前状态**：
- 当前会话上下文：~140K tokens
- 危险阈值：500K tokens（开始出现错乱）
- 已有 checkpoint 机制：light（同会话恢复）、heavy（跨会话交接）

**用户提出的增强需求**（记录到 README 作为未来工作）：
1. **自动 Checkpoint Light**
   - 每 54-60 次调用自动生成 checkpoint light
   - 主动将不常用的上下文抛弃到 checkpoint
   - 减少主上下文大小，避免接近 500K 阈值

2. **上下文分层管理**
   - 常用上下文：保留在主会话
   - 不常用上下文：存入 checkpoint，按需恢复
   - 权衡：显著降低缓存命中率，但避免上下文错乱

3. **新会话隔离策略**
   - Plan 完成后新开会话执行（planning 和 execution 分离）
   - 避免原会话上下文过重
   - Checkpoint 机制支持跨会话状态传递

**代码位置**：
- Checkpoint 管理器：`src/checkpoint/index.ts`
- Context pressure hook：`src/hooks/context-pressure/index.ts`
- 当前实现：第 560-564 行（context pressure hook 初始化）

**建议**：
- 在 README 的 "Future Enhancements" 章节记录这些需求
- 标记为 v1.2.0 或 v1.3.0 的目标功能
- 当前 v1.1.0 专注于修复已知问题

### 4. **MCP 端口策略**

**建议方案**：
- **方案 A**：移除共享逻辑，每个窗口独立启动（简单）
- **方案 B**：使用随机端口，避免冲突（复杂）
- **方案 C**：等待 OpenCode 官方 SSE 支持（长期）

**推荐**：方案 A，简单可靠

### 5. **Auto-Review 重构**

**建议方案**：
- 将 @oracle 的 review 逻辑改为主 agent 内部 checklist
- 移除 `task(subagent_type="oracle")` 调用（`todo-continuation/index.ts` 第 36 行）
- @oracle 仍保留为用户手动调用的专家 agent

### 6. **save_plan 工具 Description 优化**

**当前问题**：
- Description 不够强制性
- 缺少"禁止输出到对话框"的明确说明

**建议修改**（`src/tools/save-plan.ts` 第 8-10 行）：
```typescript
description: `Save a planner-created markdown plan to .opencode/extendai-lab/plans/ and return the real saved path.

CRITICAL: You MUST use this tool to save the plan. Do NOT output the plan content in the conversation. Do NOT ask the user where to save it.

Use this when the planner/prometheus agent has enough information to create an executable plan. Do not claim a plan was saved unless this tool returns success.`
```

**设计理念**：
- 让 AI 通过清晰的 description **决定调用工具**
- 避免反复注入路径提示（浪费 token）
- 工具 description 一次性说清楚所有规则
